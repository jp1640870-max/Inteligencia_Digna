dgx model optimizations:-------------------------||

1. Parámetros de inferencia
  OLLAMA_FLASH_ATTENTION=1	    <----->    Activa flash attention — acelera ~20-50% en contexto largo
  OLLAMA_KEEP_ALIVE=-1          <----->    Modelo permanece en memoria para siempre (nunca "cold start" mientras el servicio esté vivo)

2. Modificar Cuantización?
  gemma3:12b está en Q4_K_M (~4-bit). Si el eval rate sigue bajo, 
  probar:
    gemma3:12b-q3_K_M,

3. Mantener warm con keepalive
  Con OLLAMA_KEEP_ALIVE=-1, el modelo nunca se descarga. Si matas el servicio y lo reinicias,
  el primer request hará cold start (~4-7s) y los siguientes serán rápidos (~1-2s esperados).

Estrategia de Despliegue:
Para asegurar el éxito de la plataforma web corporativa,
arquitectura de software para aprovechar las capacidades nativas de la dgx:

Usa vLLM con soporte PagedAttention: 
No despliegar con Ollama crudo para este volumen de usuarios. 
Utiliza vLLM (que corre nativo en el stack de software de NVIDIA de tu Spark). 
Su tecnología de PagedAttention optimiza el uso de la memoria unificada del sistema reduciendo el 
desperdicio del KV Cache a casi cero.

Habilitar el modo "Thinking" con moderación: Si tus usuarios van a usar el razonamiento profundo
de Gemma 4 para tareas complejas, limita los tokens de pensamiento en la interfaz web para evitar
que un solo usuario acapare la GPU por periodos largos de tiempo.

Opcionales:
- Safety net en prompt-builder.ts para limitar a ~20 mensajes máximos
- Health-check automático en la app por si el modelo vuelve a fallar

Propuesta v1: Columna shared_context TEXT en la tabla proyectos. El usuario puede editar un bloque de texto que funciona como "instrucciones compartidas" para todos los chats del proyecto. Al enviar un mensaje en cualquier chat dentro del proyecto, se inyecta ese texto como mensaje de sistema antes del historial.
A futuro: Podríamos agregar resúmenes automáticos entre chats del proyecto, pero para empezar mantenemos algo simple y transparente para el usuario.

DB projects — resumen
Tablas
projects                   project_chats                  chats
┌─ id (PK) ───────────┐   ┌─ project_id ────────┐        ┌─ id (PK) ────┐
│  user_id → users.id  │──→│  chat_id → chats.id │←───────│  user_id     │
│  name                │   │  added_at           │        │  title       │
│  instructions        │   │  chat_title ← NUEVO │        │  messages…   │
│  created_at          │   │  PK (project_id,    │        └──────────────┘
└──────────────────────┘   │      chat_id)       │
                           └─────────────────────┘
Identificación
- Proyectos: por projects.id (UUID), siempre filtrados por user_id
- Chats: por chats.id (UUID), siempre filtrados por user_id
- Relación: project_chats con PK compuesta (project_id, chat_id)
Operaciones
Acción	SQL que ejecuta
Crear proyecto	INSERT INTO projects
Crear chat en proyecto	INSERT INTO chats + INSERT INTO project_chats (con chat_title)
Agregar chat existente	INSERT INTO project_chats (con chat_title desde chats.title)
Quitar chat del proyecto	DELETE FROM project_chats WHERE project_id=? AND chat_id=?
Eliminar proyecto	DELETE FROM project_chats WHERE project_id=? + DELETE FROM projects WHERE id=?
Eliminar chat permanente	DELETE FROM chats WHERE id=? → CASCADE a project_chats automático
Renombrar chat	UPDATE chats SET title=? + UPDATE project_chats SET chat_title=? WHERE chat_id=?
Listar chats del proyecto	SELECT c.* FROM chats c JOIN project_chats pc ON c.id=pc.chat_id WHERE pc.project_id=? — retorna {id, title, project_name, chat_title, ...}
Contar chats por proyecto	Subquery SELECT COUNT(*) FROM project_chats WHERE project_id=? (sin columna redundante)
Consistencia
- ON DELETE CASCADE en ambas FKs de project_chats — eliminar un proyecto o chat limpia las relaciones automáticamente
- chat_title se sincroniza al renombrar (vía syncProjectChatTitle())
- getUserStats() cuenta proyectos y chats con COUNT(*) en tiempo real
